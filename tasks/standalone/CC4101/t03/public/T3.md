(d) [7 Pt] Implemente la función
    ;; check-type ::   Expr TypeEnv -> Type
    que realiza la comprobación de tipos (typechecking) sobre un programa en sintaxis
    abstracta. Para ello, considere las siguientes reglas:

        Las constantes numéricas son de tipo numérico (numT).

        Las constantes booleanas ( tt y ff) son de tipo boolean (boolT).

        En el caso de la suma y la multiplicación, estas tienen tipo numérico
        (numT), si y solo si, ambos operandos tienen tipo numérico (numT). Por
        otro lado, la operación “menor que” es de tipo boolean (boolT) si ambos
        operandos son de tipo numérico.

        Un identificador obtiene su tipo a partir de la extracción de un ambiente de
        tipos (TypeEnv).

        En el caso de los condicionales (if ), la condición debe tener tipo boolT.
        Por otro lado, ambas ramas de ejecución deben tener el mismo tipo entre sí,
        siendo este el tipo que recibe el condicional completo.

        Las funciones tienen su propio tipo (arrowT) que indica el tipo del dominio
        y el tipo del codominio. En el caso del primero, las funciones son declaradas
        con el tipo del argumento, por lo que se conoce de antemano. Por otro lado,
        para el codominio se debe inferir el tipo extendiendo el ambiente de tipos
        con el argumento de la función.

        La aplicación de una función tiene el tipo del codominio si el argumento que
        se está entregando es del tipo indicado en la declaración de la función.

    Cuando las reglas anteriores no se cumplen, la función debe arrojar errores, para
    ello, siga estos ejemplos:
    >>> (check-type (add (tt) (num 50)) empty-tenv)
    check-type: Addition expects 2 numbers
    >>> (check-type (mul (num 10) (ff)) empty-tenv)
    check-type: Multiplication expects 2 numbers
    >>> (check-type (lt (num 25) (ff)) empty-tenv)
    check-type: Less-than expects 2 numbers
    >>> (check-type ( ite (num 100) (tt) (ff )) empty-tenv)
    check-type: If condition must be a boolean
    >>> (check-type ( ite (tt) (num 500) (ff)) empty-tenv)

        check-type: If branches type mismatch
        >>> (check-type (app (num 100) (num 300)) empty-tenv)
        check-type: Function application to a non-function
        >>> (check-type (app (fun 'x (numT) (mul (id x) (num 2))) (ff)) empty-tenv)
        check-type: Function argument type mismatch

(c) [6 Pt] El desafío de implementar un lenguaje con pattern matching es determinar
    si un valor calza con un patrón dado, encontrando además la asociación entre
    variable y valor. Por ejemplo, el valor (cons (1 (cons 2 3))) calza con el patrón
    (cons x y), en donde x se asocia con 1 e y con (cons 2 3). Por otro lado, el valor
    (cons 2 3) no calza con el patrón (cons x ( nil )) , pues 2 no corresponde con el
    patrón ( nil ).
    En T3.rkt se define el tipo de datos Result que permite retornar un resultado
    exitoso o un mensaje de error sin la necesidad de arrojar una excepción. De esta
    manera, se puede hacer un manejo explícito de los errores. Por ejemplo, se puede
    definir una “división segura”:
    ;; safe-div :: Number Number -> (Result String Number)
    ( define ( safe-div x y)
       (if (zero? y) ( failure "DivisionByZero") (success (/ x y))))
    Bajo esta idea, implemente la función
    ;; generate-substs :: Pattern PValue -> (Result String (Listof (Symbol *
         PValue)))
    que a partir de un patrón y un valor, retorna una lista de substituciones inducidas
    por el pattern matching o un mensaje de error si el match no es exitoso. Para
    ello, asuma que:

        nil es un valor (nilV)

        Un par construido con conz es un valor (conzV) solo si sus componentes son
        valores.

        En un patrón nunca se repite el nombre de una variable.

    >>> (generate-substs (numP 3) (numerical 3))
    (success '())
    >>> (generate-substs (numP 3) (numerical 4))
    ( failure "MatchError: given number does not match pattern")
    >>> (generate-substs (varP 'x) (numerical 3))
    (success (list (cons 'x (numV 3))))
    >>> (generate-substs (numP 3) (nilV))
    ( failure "MatchError: expected a number")
    >>> (generate-substs (nilP) (numerical 45))
    ( failure "MatchError: expected nil")
    >>> (generate-substs (conzP (varP 'x) (nilP)) (numerical 22))
    ( failure "MatchError: expected a cons constructor")
